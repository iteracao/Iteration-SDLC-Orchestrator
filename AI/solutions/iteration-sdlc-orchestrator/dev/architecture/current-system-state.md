# Current System State

## 1. System Nature

The current system is requirement-driven for change work. The main operational pipeline starts from a `Requirement`, then moves that requirement through analysis, design, planning, and implementation-related state transitions. The system is not workflow-driven in the sense of letting workflows define the business object model on their own. Instead, workflows operate on persisted domain entities and update their state.

In current implementation, the main workflow handlers all begin by loading an existing `Requirement` or a `BacklogItem` linked to a `Requirement`, validating current status, creating a `WorkflowRun`, and then updating the related domain state. The workflow is therefore a bounded processing step around an already-existing domain object, not the primary source of truth.

Agents are also bounded processors inside those workflows. They do not decide what the system is or what object should exist next on their own. The application layer gathers context, constrains file access, defines the prompt frame, parses the output, persists the reports, and applies the resulting state transitions. The agent produces structured JSON inside those boundaries.

## 2. Core Domain Model (Actual)

### Solution vs SolutionTarget

`Solution` is the logical definition. It stores the human-facing name, description, and profile code. It is used for grouping and selection in the cockpit.

`SolutionTarget` is the concrete runtime record. It stores the target storage code, repository path, main solution file, profile code, and overlay metadata. Runtime entities point to the target, not directly to the solution. In practice, `SolutionTarget` is the operational boundary for repository access and workflow execution.

The current data model still enforces one target per solution through a unique index on `SolutionTarget.SolutionId`, so the solution/target split is real in the model but not fully realized as a multi-target runtime yet.

### Requirement

`Requirement` is the central entity for change orchestration. It stores target association, optional originating backlog item, optional parent requirement, current workflow run link, title, description, type, source, status, priority, and serialized acceptance criteria / constraints.

In current usage, requirements are:

- created directly through the API and cockpit
- persisted with normalized business lifecycle states: `Pending`, `Analyze`, `Design`, `Plan`, `Implement`, `AwaitingDecision`, `Completed`, and `Cancelled`
- advanced only after the corresponding workflow run is explicitly validated
- never auto-completed after workflow validation
- used as the entry point for analyze, design, and plan workflows
- sometimes generated by analysis or implementation as derived requirements

### WorkflowRun

`WorkflowRun` is the persisted execution record for a workflow attempt. It stores:

- optional `RequirementId`
- optional `BacklogItemId`
- required `TargetSolutionId`
- `WorkflowCode`
- status enum
- current stage
- timestamps
- requested-by
- failure reason

In actual use, the workflow run is the main runtime audit record for a handler execution and is what the cockpit inspects for stage/status/failure details.

The current normalized workflow run state model is:

- `Pending`
- `Running`
- `Completed`
- `Validated`
- `Error`
- `Cancelled`

### AgentTaskRun

`AgentTaskRun` is the persisted record of the agent invocation inside a workflow run. It stores:

- `WorkflowRunId`
- `AgentCode`
- task status
- timestamps
- input payload JSON
- output payload JSON
- failure reason

Current handlers create one `AgentTaskRun` per workflow execution path. It is not a generalized multi-agent planning system today.

For `setup-solution`, the task run is preserved as deterministic workflow bookkeeping rather than an actual AI call.

### BacklogItem

`BacklogItem` is the current model for implementation slices. It stores target association, optional requirement link, optional plan workflow link, planning order, title, description, workflow code, priority, status, and timestamps.

In actual use, backlog items are primarily generated by planning and then consumed by implementation. They are ordered slices of work under a requirement, not the primary intake object. The model already includes states such as `NotImplemented`, `AwaitingValidation`, `ImplementationError`, `Validated`, and `Canceled`.

## 3. Runtime Model (Target-Scoped Execution)

Runtime operations are currently scoped to `SolutionTarget`.

The cockpit loads the main runtime state by `targetSolutionId`:

- requirements from `api/solution-targets/{targetSolutionId}/requirements`
- backlog from `api/backlog-items?targetSolutionId=...`
- workflow runs from `api/workflow-runs?targetSolutionId=...`
- open questions from `api/solution-targets/{targetSolutionId}/open-questions`
- decisions from `api/solution-targets/{targetSolutionId}/decisions`

Selected cockpit state tracks both:

- the selected `Solution`
- the selected current `SolutionTarget`

`SelectedSolutionState` stores `Current` and `CurrentTarget`, and the cockpit uses the selected target id for runtime refresh and stage actions.

Cockpit refresh is still polling-based, but it is now bounded to meaningful activity. The cockpit polls while the selected target has runs in `Pending`, `Running`, or `Completed`, performs an immediate refresh after run/validate/cancel actions, and applies snapshot-based comparison so unchanged polls do not trigger UI updates.

There is one current inconsistency: documentation browsing is still solution-based. The cockpit requests documents through `api/solutions/{solutionId}/documents`, and `SolutionDocumentsController` resolves the first target found for that solution. So runtime execution is target-scoped, but document browsing is still effectively solution-scoped / first-target-resolved.

## 4. Workflow Pipeline (Current vs Intended)

### 4.1 Implemented Pipeline

#### `analyze-request`

Trigger:

- `POST api/workflow-runs/analyze-request` with `RequirementId`

Input source:

- existing requirement
- target solution from `Requirement.TargetSolutionId`
- framework workflow config and profile rules
- repository snapshot, repository file list, repository markdown docs outside `AI/solutions/...`
- search hits and sampled files
- stable target docs under `AI/solutions/<target-storage-code>/...`

Output persistence:

- `WorkflowRun`
- `AgentTaskRun`
- `AnalysisReport`
- generated derived requirements
- generated open questions
- generated decisions
- artifacts: `analysis-request.input.json`, `analysis-report.json`
- workflow log file

Status transitions:

- workflow run: `Pending` -> `Running` -> `Completed`, `Error`, or `Cancelled`
- requirement: `Pending` -> `Analyze` when the run is created, then `Design` after validation

#### `design-solution-change`

Trigger:

- `POST api/workflow-runs/design-solution-change` with `RequirementId`

Input source:

- existing requirement in `Design`
- latest persisted analysis report linked to the requirement workflow run
- same repository/solution/framework evidence pattern used by analysis

Output persistence:

- `WorkflowRun`
- `AgentTaskRun`
- `DesignReport`
- generated open questions
- generated decisions
- artifacts: `design-request.input.json`, `design-report.json`
- workflow log file

Status transitions:

- workflow run: `Pending` -> `Running` -> `Completed`, `Error`, or `Cancelled`
- requirement: remains `Design` until the run is validated, then moves to `Plan`

#### `plan-implementation`

Trigger:

- `POST api/workflow-runs/plan-implementation` with `RequirementId`

Input source:

- existing requirement in `Plan`
- latest persisted design report
- same repository/solution/framework evidence pattern
- optional stable `design/latest-design.md` if present

Output persistence:

- `WorkflowRun`
- `AgentTaskRun`
- `PlanReport`
- generated backlog items
- generated open questions
- generated decisions
- artifacts: `plan-request.input.json`, `implementation-plan.json`

Status transitions:

- workflow run: `Pending` -> `Running` -> `Completed`, `Error`, or `Cancelled`
- requirement: remains `Plan` until the run is validated, then moves to `Implement`

#### `implement-solution-change`

Trigger:

- `POST api/workflow-runs/implement-solution-change` with `BacklogItemId`

Input source:

- existing backlog item
- linked requirement
- linked plan workflow run and plan report
- same repository/solution/framework evidence pattern
- optional stable `design/latest-design.md` and `delivery/latest-plan.md` if present

Output persistence:

- `WorkflowRun`
- `AgentTaskRun`
- `ImplementationReport`
- generated requirements
- generated open questions
- generated decisions
- artifacts: `implementation-request.input.json`, `implementation-result.json`

Status transitions:

- workflow run: `Pending` -> `Running` -> `Completed`, `Error`, or `Cancelled`
- backlog item: `NotImplemented` or `ImplementationError` -> `AwaitingValidation`, then `Validated` only after run validation
- requirement: remains `Implement` until the run is validated, then moves to `AwaitingDecision`
- on execution failure: backlog item -> `ImplementationError`; requirement does not advance

Important current reality:

- the workflow persists implementation output and advances state
- the current agent runtime is read-only and does not mutate repository files

### 4.2 Intended Pipeline (Not Fully Implemented)

The intended pipeline is clearly visible in framework config and cockpit structure:

- analyze -> design -> plan -> implement -> validate -> later stages

Current intended later stages include:

- testing / validation
- review
- delivery
- history / documentation update workflows

What is not implemented yet:

- no dedicated test/review/deliver/history runtime handlers
- no runtime handler for `test-solution-change`
- no runtime handler for `review-implementation`
- no runtime handler for `deliver-solution-change`
- no runtime handler branch for `update-solution-history`

What is partially implemented:

- validation exists as a generic workflow-run action and API endpoint, not as a dedicated workflow executor branch
- test/review/deliver lanes are shown in the cockpit as placeholders
- framework YAML defines later stages, but application/runtime execution does not yet implement them

---

### 4.3 Execution Rules (IMPORTANT)

The current code enforces these real rules:

- design requires requirement status `Design`
- planning requires requirement status `Plan`
- implementation requires requirement status `Implement`, a backlog item linked to a requirement, and a validated plan workflow run
- implementation is sequential within a plan: earlier backlog items for the same requirement/plan must already be `Validated` or `Canceled`
- a new run for the same stage is blocked while an earlier run for that stage is still `Pending`, `Running`, or `Completed`

The current code also depends on previous-stage persisted outputs:

- design reads an `AnalysisReport`
- planning reads a `DesignReport`
- implementation reads a `PlanReport`

Current no-skipping reality:

- skipping from requirement straight to design or planning is blocked by status checks
- implementation cannot start without a linked backlog item and plan report
- the next stage does not unlock on execution success alone; it unlocks only after explicit validation of the previous run

Current lack of enforcement or partial enforcement:

- validation is enforced as a generic run transition, but not modeled as its own first-class workflow type yet
- document updates suggested by agents are persisted inside reports, but not automatically applied back to stable docs
- framework-defined later stages exist, but there is no executor support for them yet

## 5. Backlog Model

Current intended behavior is that backlog is generated by planning. `StartPlanImplementationRunHandler` creates ordered `BacklogItem` records from the persisted planning output, linking them to the requirement and the plan workflow run.

`BacklogItem` is linked to `Requirement` and to the plan workflow run that created it. In current runtime use, backlog represents implementation slices under a requirement, not primary intake.

Execution is currently sequential within a single requirement/plan. The implementation handler blocks a backlog item if any earlier item in the same plan for the same requirement is not yet `Validated` or `Canceled`.

There is no richer dependency graph model yet. The current dependency model is effectively planning order plus validation gate, not explicit backlog-to-backlog dependency records.

Direct backlog API support still exists. `POST api/backlog-items` can create backlog manually, which is inconsistent with the intended model where backlog should normally be planning output.

## 6. Validation Model

Validation now exists as a generic workflow-run action, not as a dedicated workflow executor branch.

Current state behavior:

- a completed run enters `Completed`
- validating that run moves it to `Validated`
- validating a requirement workflow advances the linked requirement lifecycle according to workflow-code mapping
- validating an implementation run also advances the backlog item from `AwaitingValidation` to `Validated`

Missing piece:

- there is no first-class validation/test/review workflow handler
- there is no `WorkflowRunExecutor` branch for validation/testing/review
- the cockpit shows a validation-oriented placeholder lane, but it is not backed by a dedicated runtime slice

So validation is currently:

- modeled as a shared run transition plus lifecycle mapping
- visible in cockpit interpretation
- not yet a dedicated pipeline stage with its own executor

## 7. Setup Workflow (Current Behavior)

Setup is a real workflow today, even though it is not part of the requirement lifecycle.

Current setup behavior includes:

- run deterministically as a system workflow with no AI execution
- validate and normalize inputs
- resolve GitHub metadata
- run filesystem setup/bootstrap first
- write setup artifacts under `data/runs/<workflowRunId>/`
- persist `Solution`, `SolutionTarget`, `WorkflowRun`, and `AgentTaskRun` only after preparation succeeds
- return handoff metadata through the legacy value `NextWorkflowCode = "update-target-documentation"`, even though the implemented follow-up workflow is now `setup-documentation`

Repository handling in current code:

- `SetupSolutionHandler` currently requires the repository folder to already exist and be accessible
- `FileSystemSolutionSetupService` now aligns with that requirement and treats a missing repository root as an error
- setup initializes Git when `.git` is missing
- setup requires a GitHub repository URL and adds remote `origin` when it is not already configured
- setup creates the main `.sln` file if missing

GitHub metadata lookup:

- setup validates the GitHub repository URL
- it calls `GitHubRepositoryMetadataService`
- it resolves owner, repository name, default branch, and private/public status
- private repository lookup requires `GITHUB_TOKEN`

Overlay copy behavior:

- optional overlay/source target copy is implemented
- setup copies files and directories from the overlay source repository into the destination repository
- it skips `.git`, `.vs`, `bin`, `obj`, and junk files
- it does not overwrite existing destination files

Documentation bootstrap:

- setup creates the managed target doc root under `AI/solutions/<target-storage-code>/`
- setup seeds baseline markdown files if they do not exist
- setup preserves existing files
- setup does not bootstrap those docs from repository truth
- setup does not generate solution knowledge or interpret repository contents
- source-grounded target-documentation understanding/update now lives in the separate `setup-documentation` workflow

Persistence order:

- phase 1: validation, metadata lookup, repository/bootstrap preparation, artifact writing
- phase 2: single explicit DB transaction for `Solution`, `SolutionTarget`, `WorkflowRun`, and `AgentTaskRun`

Failure behavior:

- preparation failure leaves no committed DB state
- DB transaction failure rolls back all setup persistence and cleans up run artifacts
- successful repository/bootstrap preparation is not automatically reverted on DB failure

Current non-goals:

- setup does not invoke an AI agent, even though it still creates an `AgentTaskRun` for workflow bookkeeping consistency
- setup does not normalize the broader requirement/backlog lifecycle
- setup does not perform repository-truth documentation synthesis itself; it only prepares the repository and docs workspace for the separate `setup-documentation` run

Current limitation:

- the model enforces a single target per solution through a unique index on `SolutionTarget.SolutionId`
- update/setup flows also operate as if one target is the normal and only case

## 8. Agent Execution Model

Agents currently run through `WorkflowRunExecutor`, which dispatches by workflow code:

- `setup-documentation`
- `analyze-request`
- `design-solution-change`
- `plan-implementation`
- `implement-solution-change`

Each workflow execution persists an `AgentTaskRun` with the serialized input payload and either serialized output payload or failure reason.

The active execution model is the file-aware loop in `FileAwareAgentRunner`:

- the model is prompted with instructions and a structured prompt
- depending on workflow phase, it can request workflow input, batched repository file loading, exact-path file reads, persisted workflow output, or approved-path writes
- the orchestrator validates the requested action and file paths against the allowed action/path set for that phase
- the orchestrator injects the resulting file or tool payload back into the prompt history and continues the loop
- the loop is capped at 16 tool interactions per phase

Allowed file scope is bounded to paths that the runtime advertises up front:

- repository files discovered under `src/`
- repository markdown documentation outside `AI/solutions/...`
- stable target docs under `AI/solutions/<target-storage-code>/...`
- profile rule documents loaded from `AI/framework/profiles/...`

This is still a bounded execution model. The agent cannot browse arbitrary files. Approved-path writes now exist for the `setup-documentation` workflow so it can update managed target docs, but implementation workflows still do not mutate repository code.

## 9. Prompt Construction Model

Prompt construction is currently layered from multiple sources:

- framework agent config: prompt text and output schema loaded from `AI/framework/agents/...`
- runtime-built structured request: workflow metadata, profile summary, repository file lists, solution docs, snapshot data, search hits, sampled files, and prior-stage report data
- hardcoded orchestration layers in code: workflow discipline rules, JSON-only rules, file-read rules, and normalization logic inside the agent-host classes

This means prompt construction is partially hardcoded today. The framework files are important, but they are not the whole runtime prompt contract.

That creates a real drift risk:

- framework YAML / prompt files can say one thing
- agent-host code can layer additional behavior or assumptions
- runtime prompt shape can evolve without a corresponding framework-only change

The current system works with this arrangement, but it is an important maintenance seam and should be treated as active implementation behavior, not accidental detail.

## 10. Documentation Model (Separation)

The current codebase already distinguishes three documentation layers.

Framework docs:

- live under `AI/framework`
- define global workflow metadata, agent prompts, output schemas, and profile rules
- are generic SDLC rules and constraints, not facts about the current target implementation

Target solution docs:

- live under `AI/solutions/<target-storage-code>/`
- describe current solution behavior, architecture, local decisions, and known gaps
- are the stable documentation set for this target solution

Workflow artifacts:

- live under `data/runs/<workflowRunId>/` plus workflow log files under `data/workflow-logs/`
- store per-run input/output evidence
- are not stable architecture docs

There is still some transitional overlap:

- later workflow handlers read stable "latest" target docs such as `analysis/latest-analysis.md`, `design/latest-design.md`, and `delivery/latest-plan.md`
- agent outputs can include documentation update suggestions, but those updates are not automatically materialized into the stable docs

So the separation exists, but the workflow-to-doc maintenance loop is not fully closed yet.

## 11. Current Limitations and Known Gaps

- pipeline is not fully implemented end to end
- validation exists, but not as a first-class workflow type
- implementation does not mutate the repository
- runtime is target-scoped, but document browsing is still solution-based / first-target-resolved
- single-target-per-solution is still enforced by the current EF model and handler behavior
- prompt construction is partially hardcoded in agent-host code
- direct code and structure changes have happened outside the SDLC workflow loop
- direct backlog creation API support still exists even though planning-generated backlog is the intended normal model
- setup seeds docs but does not bootstrap them from repository truth
- later workflow handlers expect optional stable docs that setup does not scaffold
- cockpit shows later pipeline lanes that are not yet backed by runtime handlers
- setup is deterministic and transaction-safe now, but it still leaves repository-truth documentation synthesis to the separate `setup-documentation` workflow
- domain and executor-side cancellation behavior now covers `Pending`, `Running`, `Completed`, and `Error` runs, but UI/application cancellation rules still need to stay aligned with that runtime behavior

---

## 12. Current System Characterization

The current system is a transitional but already substantial SDLC orchestration platform. The core runtime model is in place: target-scoped entities, requirement-centered workflow progression, persisted workflow/report records, background execution, bounded agent processing, and a cockpit that exposes the current state clearly enough for operators to inspect and drive the implemented stages.

At the same time, the system is only partially enforcing the full intended SDLC pipeline. The implemented stages are real and operational, but validation, review, delivery, and automated stable-doc maintenance are not yet complete runtime slices. Some parts of the current behavior are also ahead of the framework consistency story, especially where code changes introduced target-scoped execution and newer cockpit behavior faster than the managed docs and workflow definitions were fully realigned.

This means the right mental model for a new engineer or agent is: the system already has a defined core SDLC architecture, but it is still in a transitional state where some rules are strongly enforced, some are only modeled in state, and some future stages are visible in config/UI before they exist as full runtime behavior. This document should therefore be read as the current operational truth, with both the confirmed direction and the confirmed gaps made explicit.
