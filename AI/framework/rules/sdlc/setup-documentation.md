# SDLC Workflow: Setup Documentation

## Classification
- Setup Documentation is a knowledge bootstrap and refresh workflow.
- Its responsibility is documentation truth maintenance, not delivery planning or implementation execution.

## Stable Documentation Scope
- The workflow may update only the managed stable solution documentation set defined by the workflow.
- It must not treat transient workflow artifacts as stable solution knowledge.
- It must not expand the stable document set beyond the managed targets for the run.

## Prohibited Side Effects
- Do not create implementation backlog.
- Do not create coding tasks, sprint slices, or delivery phases.
- Do not mutate requirement lifecycle state or requirement workflow history.
- Do not rewrite unrelated repository files.

## Traceability
- Documentation decisions must remain traceable to reviewed repository evidence.
- Stable documentation updates must reflect repository reality, not preferred future state.
- If evidence is incomplete or conflicting, preserve that uncertainty in the reasoning and outputs.

## Artifact Separation
- Repository-state, decision, and run-report artifacts support the workflow but are not stable managed documents.
- Stable docs remain the canonical knowledge outputs; artifacts explain how the workflow reached them.
